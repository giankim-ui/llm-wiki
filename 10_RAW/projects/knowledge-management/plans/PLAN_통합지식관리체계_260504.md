# PLAN — 통합 투자 지식 관리 체계

- **작성일**: 2026-05-04
- **대상 자산**: AI 채팅 기록(Claude/Gemini) + JSX/HTML 결과물(국가별) + archive 마크다운 + 캡처/스터디 자료

## 1. 현황 진단

### 1.1 현재 자산 분포 (확인된 것)    
| 자산 종류 | 위치 | 형식 | 규모 추정 |
|---|---|---|---|
| 분석 대시보드 (최종 산출물) | `S-anlyz/!Report`, `S-anlyz-kr/!Report`, `S-anlyz-jp/!Report` | `.jsx` + `.html` | 약 15+ 파일 (3개국) |
| 작업 계획/결과 문서 | `S-anlyz*/archive/*.md` | `PLAN_*.md`, `result*.md`, `research*.md` | 약 35+ 파일 |
| 핸드오프 (세션 인계) | root + `S-anlyz-jp/archive/` | `HANDOFF-*.md` | 약 10+ 파일 (root에 산재) |
| 에이전트/스킬 정의 | `S-anlyz*/.claude/agents/`, `.claude/skills/` | `.md` | 국가별 중복 보유 |
| 학습/스킬 자료 | `자료실/`, `studies/`, `blueprint/` | 캡처(.png), `.md`, `.html` | 다수 |
| **AI 채팅 기록** | **외부 (Claude.ai, Gemini, 다중 계정)** | **검색 채팅 형태** | **미수집** |

### 1.2 구조적 약점
1. **국가별 폴더 분리** → US/KR/JP 동일 분석 패턴이 3중 중복, 통합 인사이트 불가
2. **archive 산재** → plan/result/research/HANDOFF 네이밍 규칙이 폴더마다 다름 (`PLAN_xxx.md` vs `plan-xxx.md` vs `result-xxx-260503.md`)
3. **외부 채팅 미통합** → 가장 풍부한 사고 흐름이 검색 안 됨
4. **결과물(JSX) 인덱스 부재** → 어떤 종목을 분석했는지 한눈에 보는 곳이 없음
5. **메타데이터 부재** → 모든 .md가 frontmatter 없는 plain text → AI가 필터링 불가

## 2. 옵션 비교

| 옵션 | 진입 장벽 | AI 친화성 | 사용자 직관성 | 통합력 | 비용 | 결론 |
|---|---|---|---|---|---|---|
| **A. 순수 폴더 + 네이밍 규칙** | ★ 매우 낮음 | ★★★ (파일 시스템 grep) | ★★ (트리만) | ★★ | 0 | 기본기로 OK, 검색 약함 |
| **B. Obsidian 볼트** | ★★ 낮음 (1시간 학습) | ★★★★★ (.md + frontmatter + 백링크) | ★★★★★ (그래프뷰/태그/검색) | ★★★★★ | 0 (개인 무료) | **★ 추천 (1차)** |
| **C. Notion** | ★★ 낮음 | ★★ (API 필요, .md 손실) | ★★★★ | ★★★ | 무료~$10 | 채팅 누적엔 좋으나 .jsx/.html 다루기 어색 |
| **D. Supabase (DB)** | ★★★★★ 매우 높음 | ★★★★ (정형 쿼리) | ★★ (UI 직접 구축 필요) | ★★★★★ | 무료 티어 OK | 비개발자 단독 운영 부적합 — 나중 단계 |
| **E. Git + GitHub Issues/Wiki** | ★★★ | ★★★ | ★★★ | ★★★ | 0 | 이미 Git은 쓰는 중, 보조 수단으로만 |

### 2.1 추천: **B. Obsidian 볼트 (1차)** + **D. Supabase (3차 확장)**
- Obsidian은 결국 **로컬 폴더 + .md 파일**이라 현재 자산을 그대로 재사용 가능
- frontmatter(YAML)로 메타데이터를 박으면 Claude가 grep으로 정밀 필터 가능
- 채팅 기록은 export → markdown 변환 → 볼트의 `inbox/`로 투입
- 볼륨이 임계치(예: 종목 50개 이상) 넘어가면 그때 Supabase로 정형화 이관

### 왜 이렇게?
- **`10_ASSETS/`가 국가 prefix(`US-`, `KR-`, `JP-`)로 종목별 통합** → "국가별 폴더 분산" 문제 해소. 국가 비교 가능
- **번호 prefix(00/10/20…)** → Obsidian/탐색기에서 항상 같은 순서, 비개발자가 외우기 쉬움
- **`INDEX.md` 패턴** → 각 종목 폴더에 허브 노트 1개. Claude가 종목 질문 받으면 여기부터 읽음
- **`00_INBOX/`** → "지금 정리할 시간 없음, 나중에" 자료의 안전한 착륙장 (PARA의 inbox 개념)

## 4. 메타데이터 (frontmatter) 표준

모든 `.md` 상단에 다음 YAML을 강제:

```yaml
---
type: analysis | plan | result | research | chat | framework | handoff
asset: US-MSFT                # 해당 종목 (없으면 생략)
country: US | KR | JP | global
date: 2026-05-04
status: draft | active | done | archived
tags: [valuation, cyclical, dividend]
source: claude-chat | gemini-chat | self | sec-filing
related: ["[[US-MSFT/INDEX]]", "[[30_FRAMEWORKS/jp-framework]]"]
---
```

### 효과
- Claude가 `Grep -g "*.md"` + frontmatter 필드로 정밀 필터
  - 예: "2026-04 이후 작성된 KR 종목의 cyclical 태그 분석" → grep 한 방
- Obsidian Dataview 플러그인으로 사용자가 동적 표 생성 가능 (코딩 0)

## 5. 외부 채팅 기록 통합 방법

### 5.1 Claude.ai
- **현 상태**: Claude.ai에는 공식 export(JSON) 기능 있음 (Settings → Export data)
- **방법**:
  1. 월 1회 export → `00_INBOX/chats/claude/YYYY-MM/raw.json`
  2. Claude Code에 "이 raw.json을 종목/주제별 .md로 분리해줘" 요청
  3. 분리된 노트는 frontmatter `type: chat, source: claude-chat` 부여 후 `10_ASSETS/<종목>/notes/` 또는 `50_RESEARCH/`로 이동

### 5.2 Gemini
- **현 상태**: Google Takeout으로 Gemini 대화 export 가능 (HTML/JSON)
- 같은 파이프라인 적용

### 5.3 자동화 (선택)
- 일단 수동 export 권장 (월 1회, 30분 작업)
- 익숙해지면 PowerShell 스크립트로 폴더 감시 → 자동 분류 (Claude Code skill로 작성 가능)

## 6. JSX/HTML 결과물 인덱싱

- 종목 폴더 내 `INDEX.md`에 다음을 자동 채우는 템플릿 운영:
  ```markdown
  # US-MSFT 인덱스
  - **마지막 분석**: 2026-05-03
  - **대시보드**: ![[dashboard.html]]  (또는 직접 링크)
  - **핵심 노트**:
    - [[notes/2026-05-03-cyclical-check]]
    - [[notes/2026-04-20-edinet-data]]
  - **관련 채팅**: [[00_INBOX/chats/claude/2026-04/MSFT-valuation]]
  ```
- HTML 보고서는 Obsidian에서 미리보기 안 됨 → 링크만 두고 더블클릭으로 브라우저 열기 (충분)

## 8. AI(Claude) 검색 친화성 체크리스트

| 요건 | 충족 방법 |
|---|---|
| 폴더 위치만으로 type 추론 | 번호 prefix + 한 폴더 한 목적 |
| 시간 필터 | frontmatter `date` + 파일명 `_YYMMDD` 병행 |
| 종목 필터 | frontmatter `asset` + 폴더 `10_ASSETS/<티커>/` |
| 주제 필터 | frontmatter `tags` |
| 컨텍스트 폭발 방지 | `INDEX.md` 허브 → Claude가 허브만 읽고 필요한 노트만 follow |
| Plain-text 보장 | 모든 핵심 정보가 `.md` (Notion/DB와 달리 Claude가 직독 가능) |

## 9. 향후 확장 (3-6개월 뒤 검토)

### 9.1 Supabase 도입 트리거
다음 조건 **2개 이상** 충족 시 검토:
- 종목 50개 초과
- "전체 종목에서 P/E < 15 + ROE > 15%" 같은 정형 쿼리 니즈 발생
- 모바일에서 조회 필요
- 여러 사람과 공유 필요

### 9.2 Supabase 도입 시 역할 분담
- **Obsidian (1차 저장소)**: 모든 텍스트/메모/JSX/HTML — 사람의 사고 공간
- **Supabase (2차 색인)**: 종목 메타(티커, 섹터, 최근 P/E, 마지막 분석일) — 정형 쿼리 전용
- 동기화: Claude Code 스킬로 frontmatter → Supabase row 자동 upsert

### 9.3 Supabase 안 가도 되는 경우
- Obsidian Dataview만으로 80% 쿼리 충족됨. 대부분 여기서 멈춤
